Don't Let the Old World Corrupt the New

A story-first guide to the Anti-Corruption Layer — the translator that keeps your shiny modern system from absorbing the chaos of legacy code.

9 min read · Azure Architecture Patterns · Domain-Driven Design · Migration Strategy

Picture this: a brand new, beautifully planned city is built right next to a crumbling old empire. The new city has wide roads, clean water, and a logical street grid. The old empire has winding alleys, a chaotic postal system, and ancient bureaucracy that nobody fully understands.

Now the new city needs to trade with the old empire. Merchants need to cross the border daily. But here's the danger: the longer they spend in the old empire, the more they start speaking like it, thinking like it, operating like it. The chaos starts leaking back across the border. The new city slowly begins to look just as messy as the old one.

This is exactly what happens when you build a modern system that talks directly to a legacy one. The new system starts absorbing the old system's bad habits — convoluted data models, obsolete APIs, bizarre naming conventions — until your clean new code is just as tangled as the thing it was supposed to replace.

Without protection, legacy semantics bleed directly into your modern codebase — corrupting the clean design you worked so hard to build.

The core danger: When your new code starts calling CUST_REC_NO instead of customerId, and handling YYYYMMDD strings instead of proper dates, it hasn't just integrated with the legacy system — it has become part of it.

The Anti-Corruption Layer

The Anti-Corruption Layer (ACL) is a facade or adapter layer placed between two subsystems that don't share the same semantics. It translates requests that one subsystem makes to the other, ensuring that dependencies on outside subsystems don't limit an application's design.

Back to our city analogy: instead of letting merchants freely wander the old empire and come back speaking its language, you station a dedicated translator at the border gate. Every message going in gets translated into the old empire's terms. Every message coming back gets cleaned up, mapped, and handed over in the new city's language.

The merchants stay clean. The city stays modern. The chaos stays on the other side of the wall.

The ACL is the border translator. Modern system talks clean. Legacy system talks old. The ACL handles every conversion in between.

"The new city doesn't learn the old empire's language. It hires a translator and stations them at the gate."

What the translator actually does

Here's a concrete example. Say your legacy system stores orders with a field called ORD_STAT_CD where "00" means pending, "01" means shipped, and "02" means cancelled — but only on weekdays. Your modern system has an enum: OrderStatus.PENDING, OrderStatus.SHIPPED, OrderStatus.CANCELLED.

Without an ACL, your new code is littered with if (legacyCode === "00") { ... } checks everywhere. With an ACL, that mapping lives in exactly one place, and your new code never sees it.

The ACL maps each field — in both directions — so neither system ever needs to speak the other's language.

The step-by-step journey

1
Modern system makes a call in its own language. It asks for getOrder(customerId: "c-991") using clean, typed domain objects.
2
The ACL intercepts it. As a facade or adapter, it sits at the boundary and catches every outbound call before it reaches the legacy system.
3
It translates the request into legacy dialect. Maps field names, converts date formats, assembles the SOAP envelope, adds the auth tokens the old API expects.
4
The legacy system responds in its own mess. Returns CUST_REC_NO, ORD_STAT_CD: "01", and a date as "20240315". The modern system never sees any of this.
5
The ACL translates the response back. Converts, cleans, validates, and maps everything into the modern domain model before handing it back.
6
The modern system receives clean data. Gets back a proper Order object with status: "SHIPPED" and an ISO date. It has no idea what happened on the other side.
7
It logs and traces the whole thing. The ACL captures latency, errors, translation failures — giving you observability into the boundary without cluttering either system.

Not just a type converter

The ACL does more than rename fields. It's a full semantic boundary — protecting the concepts, not just the syntax.

What gets translated Legacy world Modern world
Customer identity CUST_REC_NO (numeric string) customerId (UUID)
Order status "00" / "01" / "02" PENDING / SHIPPED / CANCELLED
Dates YYYYMMDD string ISO-8601 / timestamp
API protocol SOAP / XML over HTTP REST / JSON
Error signals Magic return codes Typed exceptions / error objects
Concepts "Account" (means 12 different things) Explicit UserAccount, BillingAccount
The ACL originated in Domain-Driven Design — Eric Evans described it as the mechanism for preserving the integrity of your domain model when you're forced to integrate with foreign systems. The translation layer isn't just technical convenience; it's a design philosophy.

When should you use it?

✓ Use it when
🏗️ Migrating a large app gradually — you need to keep talking to legacy while building new
🌐 Integrating with a third-party system your team doesn't control
📐 The two systems have fundamentally different data models or business concepts
🚧 Legacy system has quality problems: convoluted schemas, obscure APIs, undocumented behaviour
✗ Skip it when
🤝 Both systems share the same semantics — the ACL would be empty boilerplate
⚡ You're retiring the legacy system very soon and the migration is trivial
📦 A simple client library solves the integration without a dedicated service
🔁 Business logic is creeping in — the ACL should only translate, not orchestrate

Things to watch out for

⏱️ Added latency
Every call passes through an extra layer. For high-throughput paths, measure the overhead and consider async / event-driven variants.
🔧 Another thing to maintain
The ACL is a service (or component) you own. Include it in your monitoring, release pipelines, and on-call rotation.
📏 Scope creep
Business logic has a habit of drifting into the ACL. Fight it. The ACL translates — it doesn't decide, orchestrate, or route business flows.
🔍 Observability is essential
Use correlation IDs and structured logging. When a translation fails, you need to know exactly which field mapping broke and why.
🛡️ Trust boundary
The ACL often bridges different trust levels. Validate and sanitise inputs coming from the legacy side before passing them into your clean domain.
🚪 Retirement plan
If the ACL is part of a migration, plan its end of life. Once the legacy system is retired, the ACL should go with it — not calcify into permanent infrastructure.

In the real world

Where you'll encounter this pattern
☁️
Azure API Management + Azure Functions — API Management handles the external facade and auth. Azure Functions implements the domain mapping. Azure Monitor traces everything. The recommended Azure reference implementation of the ACL pattern.
🔄
Strangler Fig migrations — The ACL is the essential companion to the Strangler Fig pattern. As you incrementally replace legacy functionality, the ACL keeps old and new talking until the old is gone.
📨
Event-driven ACLs — Using Azure Service Bus or Event Grid, the ACL becomes async. The modern system publishes clean events; the ACL subscribes, translates, and forwards to the legacy system without blocking the caller.
🏢
ERP integrations — Every enterprise has a SAP, Oracle, or mainframe system with its own 1990s-era data model. The ACL is standard practice when building modern microservices around these beasts.

Ambassador vs. Anti-Corruption Layer — what's the difference?

They're related but solve different problems:

AmbassadorAnti-Corruption Layer
Primary jobHandle network concerns (retries, TLS, circuit breaking)Translate semantic differences (data models, concepts)
LivesNext to your app (sidecar)Between subsystems (facade or service)
Knows aboutProtocols, connections, reliabilityDomain models, field mappings, business terms
Typical useAny outbound call to a remote serviceLegacy system integration / migration

Keeping it in your head

The new city and the old empire can coexist. Trade can flow. But you don't send your architects to live in the old empire and absorb its chaos. You station a translator at the border.

That translator speaks both languages fluently. It knows that CUST_REC_NO is what the old empire calls what you'd call a customerId. It knows the old empire writes dates backwards. It knows their status codes by heart.

But most importantly — it never lets those old-world conventions cross the border into the new city.

That's the Anti-Corruption Layer. A translator with strong opinions about what belongs in your world.
